CS 201 Mass Tester


Used to run multiple tests at once for projects
This program runs and compares multiple test cases at once for projects related to CS 201 in a Lusth-like fashion.

To Add Test Cases:
Add your .c code file into "submission". Your .c file(s) should be unique in name. Add the expected results into "expected". The file must be .txt, and must have the same name as your .c file.

To Use/Run:
Add your project code into "assign". You will need a specific argument in your Makefile. See Makefile within the root folder of the repository for an example. Your program's results will show in the folder "results".

Using Valgrind:
You have the option to use valgrind with the mass tester. Run the program with "-valgrind" in the command line argument. Valgrind will run on every test case entity.

Valgrind can return a particular error code with the argument "--error-exitcode=D", where D is an integer value. For the case of the mass tester, valgrind will return an integer value of 5 if a memory leak was detected. Otherwise it will return a 0.

RE: Mass Tester - daweil - 09-19-2018

Students will need to download submission0.gz from: http://beastie.cs.ua.edu/cs201/testing/0/submission0.tgz

Afterwords, the mass tester needs to be downloaded. This can be done directly from the page or by cloning it if git is installed:
Code:
git clone https://github.com/Dawilly/cs201tester.git

The student's code should be placed into the "assign" folder. An alternate Makefile will be needed to allow the program to latch on different files that only have the main(int argc, char** argv) function (These are the test cases). Instead of having, say assign0.c and assign0.o in your Makefile (which contain the main routine), they should be replaced with $(ARG).o and $(ARG).c. Anywhere that contains the file with the main routine (minus its file extension) should be replaced with $(ARG).

In addition, the Makefile should be outputting the compiled program as "assign". Not "assign0".

Once you extract submission0.gz, you'll need to place all the c source files (the test cases) into the "submission" folder. For every c source file, the program will be looking for the expected output file with the same filename, but not the same file extension. In other words:

If you use the test case cda-0-0.c, the program will rummage through the expected folder looking for cda-0.0.txt.

Since all the expected outcome files are in a ".expected" file extension. You can:

A.) Run a single bash shell line to rename all files with .expected to .txt below (Take note, you'll need to be in the same folder as the .expected files):

Code:
find . -name "*.expected" -exec rename -v 's/\.expected$/\.txt/i' {} \;

OR
B.) You can modify the source code to look for .expected instead of source code at Line 114:
https://github.com/Dawilly/cs201tester/blob/develop/testAssign.c#L114


Lastly, compile the mass tester:
Code:
gcc -Wall -Wextra testAssign.c -o testAssign

You can now test everything all at once. You can even test for memory leaks with -valgrind option in the command line arguments.



Since the program traverses through the directories, it can take an arbitrary amount of test cases. If every student in the class writes 2 test cases, y'all will have ~240 test cases to use against your code. You can do push requests on the repository too.

The program has been updated to allow the user to ignore successful tests with "-ignore". It also now allows the user to reset the files to their original state with "-reset".

To update the program, either redownload and recompile or use git. Make sure you have the "develop" branch checked out:
Code:
git fetch
git checkout develop
git pull
