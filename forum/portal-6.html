  Simple mass tester makefile
Posted by: magarwal - 09-19-2018, 09:11 PM - Forum: Miscellaneous - Replies (9)

I saw @dawilly's tester that required some moving around of files everytime changes were made to the project code. So I came up with some simple makefile rules to run the tests. Just put all the test files and expected outputs (such as from submission0.tgz) in a subfolder called 'testing/' (you can change this TESTDIR value) under your main project folder and then add these rules (since I'm not allowed to post the whole makefile):

1. First define all the test file names under a variable (called TESTS here) near the top of your makefile. We can use a wildcard to get all the files under our testing directory:

Code:
TESTDIR = testing/
TESTS = $(sort $(basename $(notdir $(wildcard $(TESTDIR)*.c))))

2. Make sure your .h and .o files get copied to the testing directory when you compile them:
Code:
OBJS = integer.o real.o string.o da.o cda.o stack.o queue.o
all : $(OBJS)
    cp -f *.o $(TESTDIR)
    cp -f *.h $(TESTDIR)

Then we can add a rule for compiling our tests:
Code:
$(TESTS): %: $(TESTDIR)%.c all
    gcc -Wall -Wextra -g -o $(TESTDIR)$@ $< $(OBJS)

3. Define a simple for loop for running tests:
Code:
test : all $(TESTS)
    for x in $(TESTS); do \
        echo; echo -------; echo $$x.expected; echo -------; cat $(TESTDIR)$$x.expected; \
        ./$(TESTDIR)$$x > $(TESTDIR)$$x.yours; \
        echo -------; echo $$x.yours; echo -------; cat $(TESTDIR)$$x.yours; echo -------; \
        cmp --silent $(TESTDIR)$$x.expected $(TESTDIR)$$x.yours && echo "PASSED" || echo "FAILED"; echo -------; \
    done

4. Make a valgrind rule if your machine can handle it:
Code:
valgrind: all $(TESTS)
    for x in $(TESTS); do \
        valgrind --log-file=$(TESTDIR)$$x.valgrind $(TESTDIR)$$x; \
        echo; echo -------; echo $$x.valgrind; echo -------;  cat $(TESTDIR)$$x.valgrind; echo; \
    done


That should be all. After you make/build your main modules, you can literally just run make test and it'll run all the tests and do a diff on them too. You can pipe the output to a text file with something like
Code:
make test > output.txt
or
Code:
make valgrind > output.txt

This has been tested to work wonderfully with the submission0.tgz for assign0. The output.txt will contain all the tests that were defined in the TESTS variable. Doing a search for "FAILED" in the test output.txt will show all the tests that failed.

Feel free to change this as necessary (giving proper credit where due).

Updating capacity when growing CDA
Posted by: hajames - 09-18-2018, 06:14 PM - Forum: Project 0 - No Replies

Hey everyone! I just discovered a bug that's been causing my CDA to seg fault. 

When growing your CDA, make sure to update items->capacity after growing the array and copying all your items over. If you update capacity before growing, then getCDA() will use the new capacity when generating indices and try to access indices in your original array that don't exist... which is bad. If you're experiencing segmentation faults or assertion errors when trying to display or insert after doubling, this might be your issue!

**Disclaimer: This works when using a temp array and copying items over to double. Not sure how this would affect doubling using realloc().

Thanks!

  Mass Tester
Posted by: daweil - 09-18-2018, 04:04 AM - Forum: Project 1 - Replies (4)

https://github.com/Dawilly/cs201tester


CS 201 Mass Tester


Used to run multiple tests at once for projects
This program runs and compares multiple test cases at once for projects related to CS 201 in a Lusth-like fashion.

To Add Test Cases:
Add your .c code file into "submission". Your .c file(s) should be unique in name. Add the expected results into "expected". The file must be .txt, and must have the same name as your .c file.

To Use/Run:
Add your project code into "assign". You will need a specific argument in your Makefile. See Makefile within the root folder of the repository for an example. Your program's results will show in the folder "results".

Using Valgrind:
You have the option to use valgrind with the mass tester. Run the program with "-valgrind" in the command line argument. Valgrind will run on every test case entity.

Valgrind can return a particular error code with the argument "--error-exitcode=D", where D is an integer value. For the case of the mass tester, valgrind will return an integer value of 5 if a memory leak was detected. Otherwise it will return a 0.

-------
Feel free to report any bugs: https://github.com/Dawilly/cs201tester/issues
