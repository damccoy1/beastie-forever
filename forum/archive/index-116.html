So assuming I am trying to break my own code, obviously one way to do it is to insertDA at a point that doesn't exist in the current array, or at a point that is separate from the rest of the code. In this case, should I
a) "autocorrect" the location to the back of the current array (or the front of the array if the array has 3 items and I send in insertDA(DA, sizeDA(DA)-5, x), which would put it at location -2)
b) place it in the specified spot, expanding the array as needed (say I have an array with the size of 8, and I send in insertDA(DA, 24, x), and expand my array twice to place it at position 24)
c) have a second assertion in my insertDA that makes sure that the array is placed within the appropriate bounds (so the previous example would have something like assert(sizeDA(DA) > indexDA)) or
d) assume that no code or inputs will attempt to insertDA in a spot that is separate or non-existent.

I am going to assume a, since there is no specifications, and I am supposed to test the limits of my program, but if b, c, or d is what is expected, or if there is a different expected solution entirely, that would be good to know.

08-23-2018, 02:20 PM
Excellent question!

What to do if a function receives bad input...

a) and b) are particularly bad ideas in that you would be masking an error by the user of your library. The user must have calculated a bad index, so wouldn't it be nice if the function could inform the user of that?

d) is the "C" approach: caveat emptor. It is up to the caller to make sure the input to a function is correct. This is what (in part) makes C so blindingly fast, no error checking.

I would suggest c) as an approach you should take in this class. This is what I did in my solutions. Alternatives are returning an error code (zero is usually interpreted to mean zero errors) if the original function return type is void or setting a instance variable flag to indicate the computation failed, otherwise.
